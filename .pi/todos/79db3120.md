Add first-class orchestration handling for `caller_ping` so serial/parallel runs can pause for input, surface a structured blocked state, and continue cleanly.

Typical use case example:
- `execute-plan` dispatches 4 coder subagents in parallel.
- One coder hits an ambiguity such as: "Should I preserve backward compatibility for the old config format?"
- Instead of guessing or failing hard, that child calls `caller_ping("Need decision: preserve old config support?")`.
- The parent/orchestrator surfaces the blocked task clearly, the user answers, and the task resumes or is re-dispatched so the overall run can continue.

Why this matters:
- Makes orchestration human-in-the-loop instead of fire-and-wait only.
- Improves recovery for ambiguous tasks and reduces bad autonomous guesses.

Success criteria:
- A ping from a child task is represented explicitly in orchestration state/results.
- The parent receives enough structured context to respond.
- The task can continue without losing the broader serial/parallel run context.
