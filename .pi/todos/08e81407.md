Bring pane-mode observability closer to headless parity.

Why this matters:
- Today the headless backend surfaces richer structured progress and result data than pane mode.
- Pane mode is still a major path for interactive use, so parity would improve consistency and downstream tooling.

Desired outcomes:
- Stream meaningful `onUpdate` progress from the pane backend where feasible.
- Populate pane-mode `usage` when available.
- Populate pane-mode transcript/message aggregation in a structured shape comparable to headless results.
- Keep pane TTY ergonomics intact; avoid regressing the interactive experience.

Questions / investigation:
- What minimal stream tap or transcript read path can provide structured data without breaking pane interaction?
- Should parity be exact, or 'best effort' with backend-specific caveats documented?

Success criteria:
- Callers can consume richer structured results from pane mode, not just headless mode.
- README/tool docs clearly explain any remaining pane/headless differences.
