## Problem

Coordinator agents such as `plan-refiner` depend on `subagent_run_serial` returning the child agent's full `finalMessage` so they can persist review artifacts, parse findings, and launch remediation passes.

Current behavior appears to surface only a thin preview/summary in some runs (for example just `### Status` or another truncated snippet), even when the child subagent completed successfully. That leaves coordinator workflows unable to do their core job:

- write the child's full output to an artifact file,
- parse structured findings from that output,
- and feed those findings into follow-on planner/remediator passes.

This is especially visible in review/remediation loops where the reviewer does real work, but the remediator appears idle because it never receives the actual review body.

## Goal

Make orchestration result propagation reliable for large child outputs.

The clean fix is for `subagent_run_serial` to return the full child `finalMessage` to the caller, even when the message is long. If there is an unavoidable payload-size limit, the orchestration result should include a first-class, supported way to retrieve the complete message body (for example an artifact/result handle), rather than forcing callers to scrape session state or invent prompt-level workarounds.

## Scope

- Investigate where the child `finalMessage` is being truncated, summarized, or replaced in orchestration results.
- Fix `subagent_run_serial` result propagation so callers receive the complete `results[n].finalMessage` for completed tasks.
- If full in-band return is not always feasible, add an explicit result field for complete large outputs (for example `results[n].artifactPath` or equivalent) and document the contract.
- Keep the existing lightweight commentary/preview behavior for humans if desired, but do not let that replace the machine-readable full result field.
- Add tests covering long final messages from child agents.

## Non-goals

- Do not require coordinator agents to inspect child sessions/transcripts directly.
- Do not push file-writing responsibilities down into read-only reviewer/judge agents just to work around missing result propagation.
- Do not rely on prompt hacks like asking a second subagent to echo the first subagent's output.

## Suggested implementation

1. Trace the result path from child subagent completion through orchestration aggregation to the parent tool response.
2. Identify whether truncation happens:
   - at child completion capture,
   - in orchestration aggregation,
   - in tool transport,
   - or only in displayed commentary while the raw payload is still available.
3. Ensure the raw orchestration result includes the full `finalMessage` string for each completed task.
4. If transport limits require a fallback, add a first-class full-output field/handle and guarantee it is present when truncation occurs.
5. Add tests for:
   - short final messages,
   - long multi-section markdown final messages,
   - messages containing structured findings that downstream coordinators need to parse exactly.
6. Document the contract for coordinator agents: they should consume `results[n].finalMessage` (or the documented full-output field when present), not session scraping.

## Acceptance criteria

- A child subagent that returns a long markdown final message yields a parent-visible orchestration result whose machine-readable payload contains the full message body.
- The returned payload is sufficient for a coordinator to write the child's output verbatim to disk and parse it without information loss.
- If any truncation/preview is shown in human-facing commentary, the full machine-readable payload remains available alongside it.
- If a fallback full-output handle/path is introduced, it is documented and populated deterministically whenever inline `finalMessage` cannot carry the full body.
- A regression test covers a coordinator-style scenario where a child emits a multi-finding review and the parent can recover the full review text from the orchestration result.

## Example motivating failure

A `plan-refiner` coordinator launches `plan-reviewer` successfully, but the orchestration result exposed to the parent contains only a preview like `### Status` instead of the review body. The coordinator therefore cannot write the review artifact, cannot count Error findings, and never launches the remediation edit pass even though the reviewer already did the real work.
