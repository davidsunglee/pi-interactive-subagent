## Summary
Two integration tests in `test/integration/orchestration-headless-no-mux.test.ts` have been failing on `feature/mux-free-execution` (and were also failing on parent commit `b6c37e8`, so this pre-dates the mux-free-execution review-v1 remediation):

- `subagent_serial executes through the real registered tool callback under forced headless` (line ~186)
- `subagent_serial reaches headless via selectBackend's auto fallback when no mux is present` (line ~264)

Both fail on the same assertion:

```ts
assert.ok(result.details.results[0].finalMessage.trim().length > 0)
```

The single wrapped child returns `exitCode: 0` and `isError: false`, so the serial wrapper itself is succeeding — but `finalMessage` comes back empty. The same `test-echo` agent under the bare `subagent` tool (same file, first test) produces a non-empty `finalMessage` successfully, so the child emits final text; `subagent_serial` is losing it somewhere.

Reproduction:
- Branch: `feature/mux-free-execution` (or parent `b6c37e8`)
- `which pi` must succeed
- `node --test test/integration/orchestration-headless-no-mux.test.ts` (or the two `--test-name-pattern="subagent_serial"` cases)

## Goal
Ship a fix so both `subagent_serial` integration cases pass, and the wrapper surfaces the child's final assistant text in `result.details.results[i].finalMessage` rather than dropping it.

## Problem details
- Pre-existing: identical failures reproduce on parent commit `b6c37e8` with `b6c37e8` versions of `index.ts` / `launch-spec.ts` / the test file checked out. So this is not caused by the review-v1 remediation commit (`e287448`).
- Only `subagent_serial` is affected. `subagent` (bare) and `subagent_parallel` both pass under the same test harness and same `test-echo` agent.
- The `exitCode`/`isError` fields propagate correctly. Only `finalMessage` is blank.
- Likely suspects (unverified, starting points for investigation):
  - `pi-extension/orchestration/run-serial.ts` aggregation — the field may never be copied from the backend result into the serial `TaskResult` shape.
  - `pi-extension/orchestration/tool-handlers.ts` serial tool mapping — may be dropping or renaming `finalMessage` while projecting `BackendResult` → orchestration result.
  - The headless backend's `BackendResult.finalMessage` itself — confirmed populated for the bare path via the first test, but the serial path may be reading from a different field (e.g. `transcript` last message) or an older snapshot.

## Scope
- Locate where `finalMessage` is lost on the serial path (diff the projection against the parallel path, which works).
- Fix the projection so the child's last assistant text reaches `result.details.results[i].finalMessage`.
- Keep the two existing assertions as the regression fence; add a unit test at `test/orchestration/` exercising the pure `runSerial` aggregation if the bug is in aggregation (not in tool-handler projection).

## Non-goals
- Not redesigning the serial/parallel orchestration contract.
- Not changing `BackendResult` shape or the bare `subagent` tool handler.
- Not touching the review-v1 remediation already landed on `feature/mux-free-execution` (commit `e287448`).

## Acceptance criteria
- Both `subagent_serial` integration cases pass locally with `pi` on PATH.
- A single-task `subagent_serial` call with a `test-echo`-like child returns a non-empty `finalMessage` that matches the child's last assistant text.
- Parallel behavior remains unchanged (the passing `subagent_parallel` cases still pass).
- If the fix is in orchestration (not just in the tool handler), a new unit test under `test/orchestration/run-serial.test.ts` pins the aggregation invariant without requiring `pi` on PATH.

## Open questions
- Is `finalMessage` empty because the serial wrapper picks it up before the child's terminal event lands, or because it's reading a stale snapshot from `onUpdate` buffering rather than the resolved `BackendResult`?
- Does `runParallel` avoid the bug because it awaits a different promise chain (index-order aggregation vs. serial's stop-on-error loop), or because its projection includes a field the serial path skipped?
- Should serial also return `transcript`/`usage` in `result.details.results[i]` for symmetry with the bare-subagent steer, or is `finalMessage` sufficient for this fix?
