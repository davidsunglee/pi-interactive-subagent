## Problem

Coordinator agents such as pi-config's `plan-refiner` and `code-refiner` need to call `subagent_run_serial` to dispatch their child agents. They now also declare restrictive `tools:` frontmatter to tighten their file/shell surfaces.

Current `pi-interactive-subagent` launch plumbing makes those two requirements conflict:

- `resolveDenyTools()` treats `spawning: false` as a deny-list expansion for orchestration tools. Omitting `spawning: false` merely avoids denying nested dispatch; it does not grant dispatch tools.
- `resolvePiToolsArg()` filters an agent's `tools:` value down to the seven pi builtins (`read`, `bash`, `edit`, `write`, `grep`, `find`, `ls`) and then injects only lifecycle tools (`caller_ping`, `subagent_done`).
- Therefore a coordinator `tools:` line that includes `subagent_run_serial` would currently have that token dropped before `pi --tools` is invoked.
- Because pi's `--tools` is a positive allowlist across built-in, extension, and custom tools, a restrictive coordinator allowlist without `subagent_run_serial` leaves the coordinator unable to dispatch.

Concrete downstream symptom in pi-config:

```yaml
# agent/agents/plan-refiner.md
tools: read, write, edit, grep, find, ls

# agent/agents/code-refiner.md
tools: read, write, edit, grep, find, ls, bash
```

Both coordinator prompts instruct the model to call `subagent_run_serial`, but that tool is not available under the restrictive allowlist.

## Goal

Let coordinator agents explicitly include known pi-interactive-subagent orchestration tools in their `tools:` frontmatter and have those tokens survive launch-time allowlist resolution on the pi backend.

The minimal downstream coordinator declarations should become possible:

```yaml
# plan-refiner
tools: read, write, edit, grep, find, ls, subagent_run_serial

# code-refiner
tools: read, write, edit, grep, find, ls, bash, subagent_run_serial
```

## Scope

- Update pi-backed launch allowlist resolution so recognized orchestration tool names are preserved in the `--tools` argument.
- Keep lifecycle injection (`caller_ping`, `subagent_done`) intact whenever a restrictive allowlist is emitted.
- Keep `spawning: false` as the deny mechanism for worker agents.
- Add tests covering both headless and pane pi launch paths.
- Document that coordinator agents which need nested dispatch must both:
  - run on `cli: "pi"`, and
  - explicitly list the orchestration tool(s) they need when using restrictive `tools:` frontmatter.

## Suggested implementation

1. In `pi-extension/subagents/launch-spec.ts`, add a recognized extension/orchestration tool set, likely:
   - `subagent`
   - `subagents_list`
   - `subagent_resume`
   - `subagent_run_serial`
   - `subagent_run_parallel`
   - `subagent_run_cancel`
2. Change `resolvePiToolsArg()` so it keeps tokens from both:
   - `PI_BUILTIN_TOOLS`, and
   - the recognized orchestration tool set.
3. Emit `--tools` when at least one recognized builtin or orchestration token is present, even if no builtin token is present.
4. Preserve the existing behavior that completely unknown-only declarations such as `weird, nonexistent` do not produce a lifecycle-only allowlist.
5. Add/adjust tests:
   - `test/orchestration/pi-tools-arg.test.ts`: `resolvePiToolsArg("read, subagent_run_serial")` includes `read`, `subagent_run_serial`, `caller_ping`, `subagent_done`.
   - `test/orchestration/pi-tools-arg.test.ts`: `resolvePiToolsArg("subagent_run_serial")` emits `subagent_run_serial,caller_ping,subagent_done` rather than `undefined`.
   - Pane pi launch script test includes `subagent_run_serial` when requested in `tools:`.
   - Headless pi launch test covers the spawned args/environment if an existing seam makes this cheap.
6. After the extension behavior is fixed, update downstream pi-config coordinator agent definitions to include `subagent_run_serial` explicitly.

## Non-goals

- Do not make Claude CLI coordinator sessions able to call pi orchestration tools. Coordinator agents that dispatch children still need to run on `cli: "pi"`.
- Do not add orchestration tools to worker agents that have `spawning: false`.
- Do not grant all orchestration tools by default to every restrictive agent; require explicit listing in `tools:`.
- Do not weaken lifecycle tool injection.

## Acceptance criteria

- A pi-backed subagent launched with `tools: read, subagent_run_serial` receives a `--tools` allowlist containing `read`, `subagent_run_serial`, `caller_ping`, and `subagent_done`.
- A pi-backed subagent launched with `tools: subagent_run_serial` receives a restrictive `--tools` allowlist containing `subagent_run_serial`, `caller_ping`, and `subagent_done`.
- Unknown-only `tools:` declarations still do not emit a lifecycle-only allowlist.
- Worker agents with `spawning: false` still have orchestration tools denied through `PI_DENY_TOOLS`.
- A coordinator agent with restrictive `tools:` and explicit `subagent_run_serial` can successfully call `subagent_run_serial` in an integration or smoke test.
- Documentation or README notes clearly state the contract for coordinator agents: run on pi and list required orchestration tools explicitly.
