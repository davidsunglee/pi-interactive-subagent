## Context

A recent `execute-plan` run in `/Users/david/Code/pi-config` exposed a contract gap between `pi-interactive-subagent` orchestration results and downstream parser-based workflows.

`execute-plan` Step 9 and Step 11 assume the parent orchestrator can mechanically access each child worker's full `finalMessage` and feed it into parsers such as:

- `agent/skills/execute-plan/scripts/parse-coder-report.py`
- `agent/skills/execute-plan/scripts/parse-verifier-report.py`

However, during the run, the parent session only appeared to receive the compact orchestration summary line for a coder task, e.g.:

- `The changes look correct. All 38 tests pass.`

That made it look like the coder had failed to emit the required structured report, but follow-up diagnosis showed the opposite.

## Diagnosis so far

The child coder did emit a correct structured final message.

Evidence:

- Claude session file: `~/.pi/agent/sessions/claude-code/38f05712-69ae-476f-a0b1-9a9537845798.jsonl`
- That final assistant message contains a proper report headed by:
  - `STATUS: DONE`
  - `## Completed`
  - `## Tests`
  - `## Files Changed`
  - `## Self-Review Findings`

Repo-level code inspection suggests the raw data still exists inside the orchestration stack:

- `pi-extension/subagents/backends/pane.ts`
  - `watch()` maps `sub.summary` into `BackendResult.finalMessage`
- `pi-extension/orchestration/default-deps.ts`
  - forwards `BackendResult.finalMessage` into `OrchestrationResult.finalMessage`
- `pi-extension/orchestration/tool-handlers.ts`
  - `toPublicResults()` preserves `finalMessage` in `details.results[]`
  - but the top-level blocking tool text is built by `summarize()` using only `firstLine(r.finalMessage)`

So the likely failure mode is not “coder prompt unreliability.” The more likely issue is that some parent/tool/harness path only exposes the summarized top-level text reliably, while nested `details.results[i].finalMessage` is not accessible enough for strict parser-driven consumers.

## Problem statement

We need a reliable contract for parent agents that orchestrate child subagents and then parse their reports.

Today there is evidence that:

1. the child's full structured `finalMessage` exists,
2. the orchestration tool return shape may preserve it in `details.results[]`, but
3. downstream parent agents cannot always depend on actually receiving/using that raw nested value.

That breaks or weakens workflows like `execute-plan`, which intentionally rely on strict parser-based handling rather than ad hoc interpretation.

## What to investigate

1. Reproduce the exact failure mode in a focused test or harness repro:
   - launch a coder-like child via `subagent_run_serial` / `subagent_run_parallel`
   - make it emit a known structured `finalMessage`
   - verify what the parent model/tool layer can actually consume
2. Determine where the raw child message becomes hard to access:
   - orchestration execute return shape
   - tool result serialization
   - renderer / compact summary path
   - parent tool-consumption layer in pi
3. Decide the correct long-term contract.

## Candidate fixes

Any of these may be acceptable if well-tested:

- Ensure raw `details.results[i].finalMessage` is reliably available to the parent model/tool consumer in blocking orchestration calls.
- Add a first-class artifact-backed report contract for parser-driven workers (e.g. coder/verifier report artifacts), similar to `test-runner`.
- Add a transcript-path fallback contract when raw `finalMessage` cannot be surfaced directly.
- Preserve the compact human-facing summary UI while separately guaranteeing a machine-readable raw result channel.

## Acceptance criteria

- There is a deterministic repro of the current failure mode, or a clear test proving the raw nested `finalMessage` is or is not available to parent consumers.
- The chosen fix makes parser-driven parent workflows able to recover the child's full structured report reliably, not just a one-line summary.
- The solution is covered by automated tests at the appropriate layer.
- The orchestration contract is documented clearly enough that downstream skills like `execute-plan` can depend on it without guessing.
