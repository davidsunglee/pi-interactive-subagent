Implementation notes:

- Ported the spirit of upstream `a0c089a` into `pi-extension/subagents/launch-spec.ts::resolvePiToolsArg`. The previous filter restricted requested tokens to `PI_BUILTIN_TOOLS` ∪ `SPAWNING_TOOLS`, which silently dropped custom/extension tool names declared via `tools:`. Now every non-empty requested token is preserved verbatim and merged with the `PI_LIFECYCLE_TOOLS` reservation (`caller_ping`, `subagent_done`).
- Coordinator/orchestration tokens (`subagent`, `subagents_list`, `subagent_resume`, `subagent_run_serial`, `subagent_run_parallel`, `subagent_run_cancel`) still survive because the agent must list them in its own `tools:` declaration and the new pass-through behavior preserves them. Existing tests for orchestration-token cases continue to pass.
- `PI_BUILTIN_TOOLS` set removed from `launch-spec.ts` (now only used in `test/orchestration/pi-to-claude-tools.test.ts` for an unrelated mapping check).
- New regression coverage in `test/orchestration/pi-tools-arg.test.ts`:
  - `preserves custom/extension tool names verbatim alongside reserved lifecycle tools` (the upstream-aligned scenario).
  - `emits a lifecycle-augmented allowlist even when no token matches a known builtin/orchestration tool` (replaces the previous "returns undefined when every effectiveTools entry is unmapped" expectation, which contradicted upstream `a0c089a` and the user's `tools:` declaration intent).
- Targeted verification: `node --test test/orchestration/pi-tools-arg.test.ts` → 8/8 pass.
