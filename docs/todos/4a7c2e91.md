## Problem

When `PI_SUBAGENT_MODE=headless` (or auto-fallback when no mux is available), the subagent runs **silently** â€” no live progress reaches the user. All the rich data (thinking, tool calls, tool results, token counts, cost) is captured into `BackendResult.transcript` and `BackendResult.usage` and surfaced through `onUpdate(BackendResult)` snapshots, but the orchestration layer flattens it into a one-line `"exit=0 (1234ms) â€” first lineâ€¦"` string (`pi-extension/orchestration/tool-handlers.ts:437-443`, `pi-extension/orchestration/run-serial.ts:196-207`) and nothing renders the per-event detail.

This is a regression vs. the predecessor `pi-subagent` package, which (when invoked from inside pi) rendered a collapsible block with always-visible metrics in the header â€” `formatToolCall` (`/Users/david/Code/pi-subagent/index.ts:68-135`) and `formatUsageStats` (`:42-66`) producing themed output, with "Ctrl+O to expand" interaction.

## Goal

Restore equivalent live transparency for the headless backend by rendering through pi's TUI surface (using `@mariozechner/pi-tui` `Box`/`Text`, already imported by `pi-extension/subagents/index.ts:4`). No changes needed to the backend itself â€” `headless.ts` already accumulates `transcript`/`usage` and emits per-event snapshots via `onUpdate` (`pi-extension/subagents/backends/headless.ts:430` for pi children, `:618` for Claude children). The data flows; nothing renders it.

## Non-goals

- Stderr/stdout printing from the backend (rejected â€” wrong rendering surface inside a pi parent that owns the TUI)
- Supporting non-pi parents (Claude Code etc. â€” not callers of this framework)
- Changes to the pane backend or to mux mode behavior
- Rendering in scenarios where pi has no TUI (script-mode parents, if any)
